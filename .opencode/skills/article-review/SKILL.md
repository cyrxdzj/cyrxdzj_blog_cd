# 文章审校技能

本技能用于对博客文章进行技术审校，发现其中的错误、过时信息或令人疑惑的表述。

## 输入参数

用户会提供以下之一：

- 博文 ID（如 `AICodingGetStartedByOCDS`）
- 博文标题（如 `AI Coding入门：以OpenCode+DeepSeek为例开始`）

## 执行步骤

### 第一步：查找博文

1. 读取 `src/data/index.yaml`
2. 在 `posts` 列表中匹配用户提供的内容：
   - 如果用户提供 ID，匹配 `id` 字段
   - 如果用户提供标题，匹配 `title` 字段（支持模糊匹配）
3. 如果找不到匹配的博文，告知用户并列出所有可用博文
4. 找到后，确认博文的 ID 和标题

### 第二步：读取博文内容

1. 根据博文 ID 读取文件 `src/data/posts/<id>.md`
2. 记录全文内容，准备分片

### 第三步：分片与并行审校

根据文章内容按逻辑章节分片（如：安装类文章可按"Node.js安装"、"工具A安装"、"配置类"、"使用流程类"等），每片包含完整的章节文本。

**重要规则：使用 Task 工具启动 SubAgent 进行并行审校，每个 SubAgent 负责一个分片。**

**注意：SubAgent可以同时启动多个。尽可能同时启动多个。**

启动 SubAgent 时，将分片文本写入 prompt 中，并指示 SubAgent：

1. 使用 `cyrxdzj_io_mcp_web_web_search_bing` 搜索验证不确定的技术信息
2. 使用 `cyrxdzj_io_mcp_web_web_fetch_content` 访问具体页面确认细节
3. 对每个发现的问题，返回以下信息：
   - 问题位置（章节名称或行号范围）
   - 问题描述
   - 严重程度（高/中/低）
   - 建议修正
   - 信息来源链接（如有）

### 第四步：SubAgent 故障处理

**如果某个 SubAgent 返回空结果或明显未完成（即"挂掉"），必须重新启动一个新的 SubAgent 来处理相同的分片，不得在主进程中直接处理该分片。**

判断标准：
- SubAgent 返回内容为空
- SubAgent 返回了非结构化文本而非审校结果
- SubAgent 报告了错误或超时

处理方式：重新启动一个相同配置的 SubAgent，如果连续两次失败，则在最终报告中注明该分片未能完成审校。

### 第五步：汇总报告

汇总所有 SubAgent 的发现，按以下格式输出最终报告：

```markdown
## 文章审校报告：<博文标题>

### 高严重程度问题
| # | 位置 | 问题 | 建议修正 |
|---|------|------|----------|
| ... | ... | ... | ... |

### 中严重程度问题
（同上格式）

### 低严重程度问题
（同上格式）

### 确认无误的部分
列出已验证无误的关键信息

### 总结
简要概括审校结果
```

## 审校检查要点

对每个分片，重点检查：

1. **链接准确性**：所有外部链接是否指向正确的目标
2. **技术事实**：版本号、命令、路径、配置项是否正确且未过时
3. **逻辑一致性**：因果关系是否成立，前后描述是否矛盾
4. **表述清晰度**：是否存在歧义或令人困惑的表述
5. **遗漏信息**：是否有重要步骤或注意事项被遗漏
6. **命令/代码正确性**：所有代码片段和命令是否能正确执行

## 搜索工具使用限制

对每个 SubAgent 的工具使用上限：
- 必应搜索（`cyrxdzj_io_mcp_web_web_search_bing`）最多使用 5 次
- 网页访问（`cyrxdzj_io_mcp_web_web_fetch_content`）最多使用 10 次

SubAgent 应谨慎决定访问哪些网页，优先验证最关键的技术细节。
